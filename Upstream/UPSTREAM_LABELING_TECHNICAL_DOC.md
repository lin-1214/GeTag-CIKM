# Upstream Session Labeling System - Technical Documentation

## System Overview

This document describes our **two-stage LLM-based session labeling system** that automatically generates persona/behavior labels for user behavioral sessions without predefined label spaces.

---

## 1. Problem Definition

### Input
- **Session data**: Sequential user interactions (e.g., product purchases, movie ratings, restaurant visits)
- **Item metadata**: Product names, titles, or business names
- **Item tags**: Category tags or semantic descriptors (optional but recommended)

### Output
- **Labeled sessions**: Each session assigned one or more persona labels
- **Example labels**: "Fine Dining Enthusiast", "Fast Food Lover", "Budget Conscious Shopper"

### Key Challenges
1. **No predefined label space**: Labels must be discovered from data
2. **Semantic understanding**: Requires understanding item relationships and user behavior patterns
3. **LLM output variability**: Non-deterministic generation needs stabilization
4. **Scalability**: Must handle thousands of sessions efficiently

---

## 2. System Architecture

### Two-Stage Pipeline

```
Stage 1: Cluster Generation (Label Discovery)
    Input: Sampled sessions with item metadata
    ↓
    LLM Prompt: "Analyze these sessions and create K persona categories"
    ↓
    Output: K cluster labels (e.g., ["Label_1", "Label_2", ..., "Label_K"])

Stage 2: Session Classification
    Input: All sessions + discovered labels
    ↓
    LLM Prompt: "Classify each session into one/multiple categories"
    ↓
    Output: Labeled dataset
```

---

## 3. Core Methodology

### 3.1 Stage 1: Cluster Generation

**Purpose**: Discover meaningful label space from data

**Process**:
1. **Sample sessions**: 
   - Use step-based sampling (e.g., 1 every 25 sessions)
   - Ensures diversity while maintaining efficiency
   
2. **Format sessions with context**:
   ```
   Session 123:
   - visited 'Restaurant A'
   - visited 'Restaurant B'
   
   Relevant tags: Italian, Fine Dining, Seafood, Expensive
   ```

3. **LLM clustering**:
   - Provide domain-specific instructions
   - Request exactly K persona-style labels
   - Include tag context for semantic grounding

**Key Parameters**:
- `T_CLUSTERS`: Number of clusters to generate (default: 20)
- `SAMPLE_STEP`: Sampling interval (default: 25)
- `INCLUDE_TAG`: Tag type ("base" or "native")

**Example Prompt Structure**:
```
Task: Analyze restaurant visiting sessions and create 20 user personas.

Guidelines:
- Use behavior-centric labels (e.g., "Fine Dining Enthusiast")
- Avoid specific restaurant names
- Keep labels concise (2-4 words)
- Make labels mutually exclusive

Sessions:
[... formatted sessions with tags ...]

Output Format:
{
  "Categories": {
    "Category 1": "Fine Dining Enthusiast",
    "Category 2": "Fast Food Lover",
    ...
  }
}
```

---

### 3.2 Stage 2: Session Classification

**Purpose**: Assign discovered labels to all sessions

**Process**:
1. **Batch processing**:
   - Process sessions in batches (e.g., 4 per batch)
   - Maintains efficiency for large datasets

2. **Classification modes**:
   - **Single-label**: Each session → 1 label
   - **Multi-label**: Each session → 1~N labels (e.g., 1-5)

3. **LLM classification**:
   - Provide discovered labels as reference
   - Include analysis dimensions (products, patterns, consistency)
   - Request structured JSON output

**Key Parameters**:
- `BATCH_SIZE`: Sessions per batch (default: 4)
- `MULTI_LABEL`: Enable multi-label mode (default: True)
- `MIN_LABELS_PER_SESSION`: Minimum labels (default: 1)
- `MAX_LABELS_PER_SESSION`: Maximum labels (default: 5)

**Example Prompt Structure**:
```
Task: Classify each session using these categories:
["Fine Dining Enthusiast", "Fast Food Lover", ...]

Analyze based on:
- Cuisine types and restaurant categories
- Dining preferences (fine dining vs. casual)
- Price ranges and quality levels
- Visiting patterns

Sessions:
### SESSION 1:
- visited 'Buffet Royal'
- visited 'Golden Corral'

### SESSION 2:
[...]

Output Format (multi-label):
[
  {"predicted_classes": ["Buffet Enthusiast", "Budget Diner"]},
  {"predicted_classes": ["Fine Dining Enthusiast"]}
]
```

### 3.3 Tag-Aware Context Enhancement

**Motivation**: Leverage domain knowledge to improve semantic understanding

**Implementation**:

1. **Tag Sources**:
   - **Native tags**: Original category labels (e.g., "Restaurants", "Italian")
   - **Base tags**: Expanded semantic tags generated by GPT (e.g., "fine dining", "romantic atmosphere")

2. **Mapping Pipeline**:
   ```
   CSV: business_id or product_name
     ↓
   Mapping: business_id → internal_id
     ↓
   Tags: internal_id → tag_list
     ↓
   Session: Aggregate tags from all items
   ```

3. **Tag Aggregation**:
   - For each item in session: extract top-3 tags
   - Concatenate tags from all items
   - Include in prompt as "Relevant tags: ..."

**Benefits**:
- Provides semantic hints to LLM
- Grounds clustering in domain knowledge
- Improves label quality and specificity

---

### 3.4 Multi-Beam Ensemble Strategy

**Motivation**: Reduce LLM output variance and improve label stability

**Problem**: 
- LLM outputs vary across runs even with same input
- Single-run results may be suboptimal or inconsistent

**Solution**: Run classification K times and vote

**Algorithm**:

```python
# Parameters
K = 5                    # Number of beams
temperature = 1.5        # High for diversity
min_frequency = 0.40     # Minimum vote threshold (40%)
top_N = 5               # Maximum labels per session

# Process
For each session:
    labels_votes = []
    
    # Step 1: Run K beams
    For beam in [1..K]:
        labels_k = LLM_classify(session, temp=1.5, seed=None)
        labels_votes.append(labels_k)
    
    # Step 2: Count votes
    label_frequency = Counter()
    For labels in labels_votes:
        For label in labels:
            label_frequency[label] += 1
    
    # Step 3: Filter by frequency
    min_count = ceil(min_frequency * K)  # e.g., 0.40 * 5 = 2
    qualified = [label for label, count in label_frequency.items() 
                 if count >= min_count]
    
    # Step 4: Select top-N by frequency
    qualified.sort(key=lambda x: (-label_frequency[x], x))
    final_labels = qualified[:top_N]
    
    Return final_labels
```

**Key Design Choices**:

1. **High Temperature (1.5)**:
   - Encourages diversity across beams
   - Prevents all beams from producing identical output

2. **No Fixed Seed**:
   - Ensures different outputs per beam
   - Explores label space more thoroughly

3. **Frequency Threshold (40%)**:
   - Labels must appear in ≥40% of beams
   - Filters unstable/random labels

4. **"Rather Less Than More" Principle**:
   - Only include labels meeting threshold
   - Don't pad to top_N if insufficient consensus
   - Example: If only 2 labels meet threshold, output 2 (not 5)

**Example**:
```
Session 123 classification across 5 beams:

Beam 1: ["Fine Dining", "Italian Lover", "Seafood Fan"]
Beam 2: ["Fine Dining", "Expensive Taste", "Italian Lover"]
Beam 3: ["Fine Dining", "Italian Lover", "Wine Enthusiast"]
Beam 4: ["Fine Dining", "Italian Lover"]
Beam 5: ["Fine Dining", "Romantic Diner", "Italian Lover"]

Vote counts:
- Fine Dining: 5/5 (100%)
- Italian Lover: 5/5 (100%)
- Expensive Taste: 1/5 (20%)
- Seafood Fan: 1/5 (20%)
- Wine Enthusiast: 1/5 (20%)
- Romantic Diner: 1/5 (20%)

Threshold: 2 votes (40% of 5)

Final labels: ["Fine Dining", "Italian Lover"]  ← Only 2 qualified
```

### 4.2 Data Flow

```
1. Load CSV
   ↓
2. Session Sampling (Stage 1)
   - Step-based: sample 1 every SAMPLE_STEP
   ↓
3. Session Formatting
   - Map ID → Name → Tags
   - Format: "- visited 'Name'\nRelevant tags: tag1, tag2, ..."
   ↓
4. Cluster Generation
   - LLM generates K labels
   - Extract labels from JSON response
   ↓
5. Full Session Classification (Stage 2)
   - Continuous slice: START_INDEX ~ (START_INDEX + SAMPLE_SIZE)
   - Process in batches
   ↓
6. Multi-Beam Classification (Optional)
   - Run classification K times per session
   - Frequency voting
   ↓
7. Output
   - CSV: session_data + Class column
   - JSON: prompts, statistics
```

## 6. Design Rationale

### 6.1 Why Two-Stage?

**Alternative**: Direct end-to-end classification
- **Problem**: Requires predefined label space
- **Our approach**: Discover labels from data first

**Benefits**:
- Adaptable to any domain without manual labeling
- Labels emerge from actual user behavior patterns
- Can be iteratively refined

### 6.2 Why Tag-Aware Context?

**Alternative**: Only use item names
- **Problem**: Limited semantic information
- **Example**: "ABC Restaurant" tells us nothing

**Our approach**: Include category tags
- **Example**: "ABC Restaurant" + "Italian, Fine Dining, Expensive"
- **Benefit**: LLM can ground clustering in domain knowledge

### 6.3 Why Multi-Beam?

**Alternative**: Single-run classification
- **Problem**: LLM output varies significantly across runs
- **Example**: Same session may get different labels

**Our approach**: Vote across K runs
- **Benefit**: Stable, high-consensus labels
- **Trade-off**: K times more LLM calls

### 6.4 Why High Temperature in Multi-Beam?

**Alternative**: Use low temperature (e.g., 0.0)
- **Problem**: All beams produce identical output
- **Result**: Voting is meaningless

**Our approach**: High temperature (1.5)
- **Benefit**: Explore different valid labelings
- **Result**: Meaningful voting and consensus

---

## 7. Limitations and Considerations

### 7.1 Computational Cost

**Multi-beam**:
- Cost: K times more LLM inference
- Example: 5 beams × 1000 sessions = 5000 LLM calls
- Mitigation: Use only when label stability is critical

### 7.2 Label Quality Dependency

**On Cluster Generation**:
- Poor cluster labels → Poor classification
- Solution: Iterate or manually review cluster labels

**On Tag Quality**:
- Missing/poor tags → Less informative context
- Solution: Use base tags (GPT-generated) for richer semantics

### 7.3 Scalability

**Current approach**:
- Batch size = 4 sessions per LLM call
- For 10k sessions: 2500 LLM calls (single-beam)

**Scaling strategies**:
- Increase batch size (may reduce quality)
- Use distillation (train classifier on LLM outputs)
- Parallel processing across GPUs

### 7.4 Multi-Beam Not Always Better

**Observations**:
- Sometimes similar results to single-run
- Depends on: task difficulty, LLM quality, label space size

**When to use**:
- High-stakes applications (label consistency crucial)
- Ambiguous sessions (multiple valid interpretations)
- Small label spaces (easier to reach consensus)
